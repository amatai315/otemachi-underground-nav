# PLAN

## 目的

`otemachi-underground-nav-requirements.md` に基づき、大手町地下移動ナビ PoC を一気通貫で実装する。
スマホ縦画面前提の Three.js 3D アプリで、大手町駅の地下コンコースを表示し、実データ由来の
POI 名称(改札・出口)から出発地/目的地を選ぶと、その駅構内ネットワークをそのままグラフとした
ダイクストラ法で最短経路を計算し、3D空間上にハイライト表示する。カメラはタッチ操作(ドラッグ回転・
ピンチズーム)で自由に動かせる。Vite + Vanilla JS + Three.js で実装し、GitHub Pages に静的サイトと
してデプロイできるようにする。

## 現状の調査結果

- `raw-data/raw-datatokyo-indoor-map.zip.zip` は「東京駅周辺屋内地図オープンデータ(Shapefile)」
  一式。ファイル名は Shift-JIS エンコードで、展開時に手動デコードが必要だった。18の建物フォルダ
  + トップレベルの `nw/` フォルダという構成。作業用に `raw-data/extracted/` へ展開済み。
- 「大手町」という名前のフォルダは存在しない。大手町駅のデータは `2.地下鉄_公共通路/B1/` 配下に
  `ChiyodaOtemati_B1_*`、`MaruOtemati_B1_*`、`MitaOtemati_B1_*` として、また同じ大手町駅でも
  `Hanzo_B1_*`(半蔵門線)・`Tozai_B1_*`/`Tozai_B2_*`(東西線)としてファイル名に「Otemati」が
  付かない形で存在する。
- `nw/Tokyo_node.shp` (3368点: `node_id, lat, lon, ordinal, in_out, link1_id..link9_id`) と
  `nw/Tokyo_Link.shp` (3798本: `link_id, start_id, end_id, distance, ...`) が、要件定義の
  データ設計で言及されている歩行空間ネットワーク。`distance` は実測メートルで、そのまま
  ダイクストラの重みとして使える。`ordinal` はフロア階層を表す(例: `-1.0`=B1、`1.0`=1F、
  0.5刻みは中2階などの踊り場)。
- ノード自体には名称が無い。POI名称は各建物フォルダの `Opening.shp`(改札。例: `大手門方面改札`)
  と、`Facility.shp` のうち `category == 'F108'`(実際の出口番号。例: 千代田線大手町駅の
  `C1`〜`C14`、丸ノ内線大手町駅の `A1`〜`A5`)に入っている。実測したところ、名称付きPOIから
  最寄りノードまでの距離はほぼ0〜5m、最悪でも約18m(ChiyodaOtemati データにて)。
- DBFのテキスト属性はUTF-8だが `.cpg` ファイルが同梱されておらず、GDAL/geopandas がデフォルトで
  CP932と誤判定し日本語が文字化けする。`gpd.read_file` には必ず `encoding='utf-8'` を明示する
  必要がある。
- 元のCRSはEPSG:6668(JGD2011 経緯度)。標高/Z値のフィールドはどこにも存在しない(2.5Dデータ)ため、
  フロア高さは `ordinal` から合成する必要がある。
- 大手町駅の基準点(ChiyodaOtemati の改札ジオメトリの重心)からの実距離で範囲を検証したところ、
  450mを閾値にすると綺麗な境界ができる: 5路線すべて(千代田線・丸ノ内線・都営三田線・半蔵門線・
  東西線)の大手町駅コンコースと、直結する地下階(オーテモリ、イーヨビル、三菱UFJ信託銀行本店ビル、
  丸の内オアゾ、新丸の内ビル)が含まれ、東京駅本体のコンコース・丸ビル・KITTE・二重橋前駅など
  (いずれも487m以上)は除外される。歩行者ネットワークはこれら直結の地下建物にも実際に伸びている
  ことを確認済み(各68〜137ノード)ので、単なる見た目の飾りではなく機能的にも意味がある。
- 接続性チェック: `Tokyo_node`/`Tokyo_Link` を450m圏内(エッジの両端が圏内)で絞ると776ノード、
  うち762ノードが単一の連結成分を構成する。残りは境界付近の孤立した小島(1〜5ノード)で、
  最大連結成分のみを採用すれば問題ない。
- Node/Python環境: `geopandas`/`fiona`/`pyproj`/`shapely` は未インストールだったため pip で
  導入済み。Vite アプリ用に Node v24 / npm v11 も利用可能。

## 決定事項

1. **データパイプラインは一度きりのローカルPythonスクリプト**(`scripts/build_geojson.py`)とし、
   手動実行する(Viteビルドには組み込まない。要件定義の「ビルド前のローカル作業」の通り)。
   `raw-data/extracted/` を読み込み、`web/src/data/` に成果物を書き出す。
2. **空間的な抽出範囲**: 大手町の基準点(`ChiyodaOtemati_B1_Opening` の改札ジオメトリの重心)から
   半径450m。建物フォルダ単位ではなくフィーチャ単位で適用するため、境界付近(東京駅西側の通路など)
   も手動選定なしに自然に扱える。
3. **CRS**: EPSG:6668 から EPSG:6677(JGD2011 平面直角座標系 IX系、東京をカバー)へ再投影し、
   メートル単位の正確な座標にした上で、基準点をローカル原点(0,0)に平行移動する。
4. **フロア高さ**: `y = ordinal * 4.0` メートル。元データに標高情報が無いための視覚的な近似である
   ことをREADMEに明記する。
5. **グラフ構築**: `Tokyo_node`/`Tokyo_Link` を450m圏内で絞り、最大連結成分のみを採用し、
   `distance` をエッジ重みとして使う。
6. **POI命名**: 各ノードについて、同じフロアの `Opening`(改札)または `Facility`(category=F108、
   出口)のうち25m以内(調査で安全と確認した許容誤差)で最も近い点を探し、見つかればその名称を
   ノードに付与し「選択可能」とマークする。UIの出発地/目的地リストにはこの名称付き(選択可能)
   ノードのみを表示し、実在データ由来のPOI名称を使うという要件を満たす。名称の無いノードは
   経由点としてグラフに残す。
7. **出力フォーマット**(要件定義のデータ設計に対応):
   - `web/src/data/floors.geojson` — 押し出し用の通路・部屋ポリゴン(対象範囲内の全建物・全フロアの
     `Space.shp` を統合)。プロパティは `{floorOrdinal, toll, source}`(有料/改札内エリアか
     無料通路かの簡易スタイリング用)。
   - `web/src/data/graph.json` — `{nodes: [{id, x, y, z, floorOrdinal}], edges: [{from, to,
     weight}]}`。
   - `web/src/data/pois.json` — `[{id, nodeId, name, floorOrdinal, x, y, z}]`。ユーザーが選択
     できる出発地/目的地リスト。
8. **描画方針**: 各 `Space` ポリゴンを `THREE.ExtrudeGeometry` で `y=floorOrdinal*4.0` から
   `y=floorOrdinal*4.0+3.0` まで押し出したフラットな柱状形状として表示する(ポリゴンの輪から
   `Shape` を生成)。壁専用メッシュは別途作らず、押し出した通路・部屋のブロックそのものを壁+床の
   表現として扱う(要件を満たしつつ複雑さを増やさないため)。改札内(有料)エリアと無料通路は
   色分けする。
9. **経路ハイライト**: 経路をノードの実座標を辿る `THREE.Line`(または Tube)として描画する
   (階段やエレベーターで実際に上下する様子が見えるように)。通過したフロアポリゴンの色変更までは
   スコープ外とし、ライン表示のみで要件(「色変更 or ライン表示」)を満たす。
10. **カメラ操作**: `three/examples/jsm/controls/OrbitControls.js` を使用。1本指ドラッグ回転・
    2本指ピンチズームをネイティブにサポートしている。パン操作は無効化する(狭いスマホ画面での
    誤操作を避けるため、また要件上も不要)。
11. **UI**: 画面上部または下部に固定のコントロールパネルを置き(3Dビューを広く取れる方を採用)、
    出発地/目的地をタップで選ぶ2つのリストと「経路探索」ボタンを配置。タップしやすい大きめの
    ターゲットサイズ。プレーンCSS、縦画面幅375〜430pxを基準とし、PC向けのブレークポイントは
    設けない。
12. **GitHub Pages のbase path**: `vite.config.js` で `base: './'`(相対パス)を使う。リポジトリ名
    を固定でハードコードする方式ではないため、今の時点でリポジトリ名を確認する必要が無い(単一
    ページアプリでクライアントサイドルーターも無いため、相対パスで問題なく動作する)。
13. **リポジトリの取り扱い**: `raw-data/`(`extracted/` を含む)はgit管理外(.gitignore)とし、
    `web/src/data/` 配下の変換済みGeoJSON/JSONのみをコミットする。元データのライセンスが
    単純な出典表示のみで再配布まで許可しているか明確でないため、安全側に倒す。READMEにはデータ
    出典のクレジットを記載する。

## スコープ

対象に含むもの:
- `scripts/build_geojson.py`: Shapefile → 抽出・再投影済みGeoJSON/JSONへの変換パイプライン
  (ユーザー依頼の手順1〜3に対応)。
- `web/`: Vite + Vanilla JS + Three.js アプリ — 3Dフロア描画、出発地/目的地選択UI、
  ダイクストラ経路探索+ハイライト、タッチカメラ操作、縦画面専用レイアウト(手順4に対応)。
- `vite.config.js` の GitHub Pages 用 base path 設定(手順5に対応)。
- `README.md`: ローカルでの動かし方 + GitHub Pagesへのデプロイ手順(手順6に対応)。
- `raw-data/` 用の `.gitignore`。

対象外(要件定義の「Out of Scope」通り、変更なし):
- 実測位(GPS/PDR/画像照合)。
- PC/タブレット向けレスポンシブ対応。
- バリアフリールート等の重み付け。
- サーバーサイド処理・API。

## 手順

1. **データパイプライン** (`scripts/build_geojson.py`):
   a. `nw/Tokyo_node.shp` + `nw/Tokyo_Link.shp` を読み込み(encoding='utf-8')、大手町の基準点を
      算出し、450m圏内に絞り込み、最大連結成分のみを残す。
   b. 対象範囲内の全建物・全フロアフォルダから `Space.shp` を読み込み、EPSG:6677へ再投影、
      投影後のメートル単位で同じ450m圏内にフィーチャ単位で絞り込み、ローカル原点へ平行移動して
      `floors.geojson` を出力する。
   c. 同じ対象フォルダから `Opening.shp` と `Facility.shp`(category=F108)を読み込み、同フロアの
      最寄りグラフノードへ25m以内でスナップし、`pois.json` を書き出し `graph.json` に注釈を付ける。
   d. `graph.json` を書き出す(ローカルx/y/z付きノード、重み付きエッジ)。
   e. サマリー(フィーチャ数、接続性チェック、スナップに失敗したPOI等)を出力し、パイプラインの
      問題をすぐに把握できるようにする。
2. **Viteアプリの雛形** (`web/`): Vite vanilla-js テンプレート、Three.js依存関係、プロジェクト構成
   (`src/main.js`, `src/scene.js`, `src/routing.js`, `src/ui.js`)。
3. **3Dシーン**: `floors.geojson` を読み込み、フロア/改札内外で色分けして押し出し、タッチ操作に
   適した設定の `OrbitControls`、縦画面ビューポートサイズ調整。
4. **経路探索UI**: `pois.json` からフロアごとにグループ化したPOIリストを描画し、「経路探索」
   ボタンを `graph.json` 上のダイクストラ実行に接続し、結果経路をシーン上でハイライトする。
5. **モバイルUIの仕上げ**: 固定コントロールパネル(上部または下部)、大きめのタップターゲット、
   縦画面優先CSS、viewport metaタグ、タッチジェスチャーの確認。
6. **ビルド設定**: `vite.config.js` に `base: './'` を設定。サブパス配下で配信した場合に成果物が
   正しく動作するか確認する(`vite preview` またはサブディレクトリからのローカル静的サーバーで
   模擬)。
7. **README**: ローカル開発手順(`npm install`, `npm run dev`)、ビルド手順(`npm run build`)、
   GitHub Pagesへのデプロイ手順(`gh-pages` npmパッケージ、または手動での `docs/` フォルダ公開など、
   現時点でgitリモートが無いことを踏まえてより簡単な方を採用)、データ出典のクレジット表記。
8. **動作確認**: 開発サーバーを起動し、実際のフローを確認する — 出発地/目的地を選ぶ、経路
   ハイライトが表示される、タッチ(実機が無ければDevToolsのタッチエミュレーション)でドラッグ回転・
   ピンチズームができる。

## 検証

- パイプライン: `scripts/build_geojson.py` を実行し、出力されるサマリー統計(ノード/エッジ数、
  POIスナップ失敗、連結成分サイズ)が上記調査結果(約762ノード、想定外の欠落が無いこと)と
  一致することを目視確認する。
- アプリ: `npm run dev` を実行し、ブラウザ(スマホ縦画面を模して約390×844にリサイズ)で確認する —
  3Dシーンが描画される、POIリストに実在の日本語名称が入っている、出発地・目的地を選んで
  「経路探索」をタップすると連続した経路がハイライトされる、ドラッグでカメラが回転する、
  ピンチ(またはDevToolsでのCtrl+スクロール模擬)でズームできる。
- ビルド: `npm run build && npm run preview` を実行し、相対base pathでアセットが正しく読み込まれる
  ことを確認する。
- このPoCでは自動テストスイートは用意しない(ハッカソンのプロトタイプというスコープに合わせる)。
  実装後のコードレビューが、自動テストの代わりの品質ゲートとなる。

## リスク

- **フロア高さの近似**(`ordinal * 4.0`)により、実際のスラブ高さが異なる建物同士で見た目上の
  重なりが不自然になる可能性がある。PoCとしては許容範囲であり、READMEに既知の簡略化として
  明記する。
- **POIスナップ許容誤差(25m)** が、密集したエリアで誤った近傍ノードに吸着させてしまう可能性が
  ある。フロアごとに最も近いノードを選ぶこと、およびパイプラインのサマリー出力で許容誤差を
  超えたPOIを検出可能にすることで軽減する。
- **F108/toll以外の分類コードの意味は不明**(データセットに凡例ファイルが同梱されていない)ため、
  スタイリングは改札内/無料通路の区別にとどまり、より細かい部屋種別の区別はしない。要件が
  壁・床・通路の基本的な表現のみを求めているため許容範囲。
- **GitHub Pagesへのデプロイ手順はこのセッション内では未検証**(まだGitHubリモートが設定されて
  いないため)。READMEに手順は正確に記載するが、実際の公開はユーザーがリポジトリを作成・push
  した後に確認してもらう必要がある。
